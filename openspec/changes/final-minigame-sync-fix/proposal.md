## Why

The current minigame synchronization logic still has edge cases where rival identification fails or the Host/Client handshake timing is inconsistent. This final fix introduces a reliable way to retrieve the rival's username from the `RemotePlayer` component and enforces an immediate local UI response for the Host, while maintaining a passive state for the Client until the network broadcast arrives.

## What Changes

- **Reliable Rival Identification**: Update `Player.cs` to retrieve the opponent's username directly from the `RemotePlayer` component during collision.
- **Deterministic Host-Initiated UI**: The Host (smaller lexicographical name) now activates the `MinijocUIManager` locally *immediately* before sending the broadcast, reducing latency for the initiator.
- **Strict Client-Side Waiting**: The Client (larger lexicographical name) is strictly forbidden from opening the UI locally and must yield execution after freezing movement, waiting for the remote command.
- **Synchronized Game Selection**: Ensures both participants use the exact same `gameIndex` generated by the authoritative Host.

## Capabilities

### New Capabilities
- `reliable-rival-discovery`: Protocol for extracting network-synchronized identities from collision partners.

### Modified Capabilities
- `combat-host-selection`: Refine the leader-follower timing to favor immediate Host responsiveness while maintaining Client passivity.

## Impact

- `Player.cs`: Core collision handler update.
- `RemotePlayer.cs`: Username must be accessible.
- `MinijocUIManager.cs`: Must be discoverable via `Resources.FindObjectsOfTypeAll`.
- `MenuManager.cs`: Receiver logic remains as the sole trigger for the Client.
